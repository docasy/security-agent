二、双路径工作流 — 一条代码，两种模式

路径 A — 告警分析（SOC 场景）

被动接收安全告警 → RAG 知识增强 → ReACT 研判 → 响应计划 → 报告

RAG 知识检索 → ReACT 威胁分析 → 响应计划 → 生成报告
数据流:
alert_data → ChromaDB 检索 TOP-3 相关知识 → 注入 ReACT Agent 上下文 → VT 查 IP/哈希 → CVE 搜漏洞 → 提取 IOC → 研判结论 (FP/Suspicious/Confirmed) → P0-P3 响应时间线 → Markdown 结构化报告
VirusTotal NVD CVE ATT&CK 映射 ReACT 范式

路径 B — 渗透测试（红队场景）

主动侦察目标 → MCP 工具链编排 → 漏洞关联 → 响应计划 → 报告

nmap 端口扫描 → nmap 服务检测 → whatweb 指纹
nmap 漏洞扫描 → exploitdb 搜索 → 响应计划 → 综合报告
MCP 调用链:
target → MCPClientManager 连接 3 个 Server 子进程 → scan_ports → detect_services → vuln_scan → fingerprint_web (如果发现 Web 端口) → search_exploits (按版本关键词) → LLM 综合报告 (风险矩阵 + 修复路线图)
MCP 协议 进程隔离 nmap whatweb exploitdb 2026 核心考点
⬆ 两条路径在「响应计划」节点汇合，共享报告生成流程

三、MCP 渗透测试工具链 — 2026 面试核心

🔍 nmap MCP Server

网络扫描器 · 渗透测试第一步

  • scan_ports — TCP 端口扫描 (1-1000)
  • detect_services — 服务版本检测 (-sV)
  • os_fingerprint — OS 识别 (-O)
  • vuln_scan — NSE 漏洞扫描

mock 模式内置完整扫描数据

🌐 whatweb MCP Server

Web 指纹识别 · 技术栈侦察

  • fingerprint_web — 全栈技术识别
  • detect_cms — CMS + 插件检测
  • find_login_pages — 管理入口发现
  • check_technologies — 版本精准验证

mock 数据模拟 WordPress 企业站侦察

💣 exploitdb MCP Server

漏洞利用数据库 · 漏洞关联

  • search_exploits — 关键词搜索
  • get_exploit_info — EDB-ID 详情
  • search_by_cve — CVE 精准关联
  • list_recent_exploits — 最新威胁态势

内置 Log4Shell/ProxyShell/MOVEit 等经典漏洞

MCP 协议 vs 传统 Function Calling — 为什么选 MCP？

维度传统 Function CallingMCP 协议（本项目）
工具定义位置 写死在 Agent 代码里 MCP Server 独立定义，Agent 动态发现
进程模型 同进程调用 独立子进程，stdio 通信
跨框架复用 仅限当前框架生态 任何支持 MCP 的 Agent 都能用
安全隔离 无隔离（共享内存） 进程级隔离，可限制权限
新增工具 改 Agent 代码 + 重新部署 写 MCP Server → 注册 → 自动发现
类比 设备专用充电线 USB-C 通用接口

四、一次渗透测试的数据流追踪

以 POST /pentest {"target": "192.168.1.1"} 为例，追踪数据在六层架构中的流转：

1. API 层 接收请求 → Pydantic 验证 → 调用 workflow.ainvoke({task_type:"pentest", target:"192.168.1.1"}) 2. 编排层 route_entry 检测 task_type="pentest" → 路由到 run_pentest 节点 3. Agent 层 PentestAgent.ensure_mcp() → MCPClientManager.start_all() → 连接 3 个 Server 子进程 4. 工具层 call_tool("nmap","scan_ports",{target,ports}) → nmap Server 子进程执行 → 返回端口列表 5. 工具层 call_tool("nmap","detect_services",{target}) → 返回精确版本号 6. 工具层 call_tool("nmap","vuln_scan",{target,cat:"vuln"}) → 返回 NSE 漏洞检测结果 7. 工具层 检测到 80/tcp 开放 → call_tool("whatweb","fingerprint_web",{url}) → Web 技术栈 8. Agent 层 _extract_keywords("Apache 2.4.41\nMySQL 5.7.38\n...") → 提取版本关键词 9. 工具层 call_tool("exploitdb","search_exploits",{query:"Apache 2.4.41"}) → 搜索已知漏洞利用 10.Agent 层 generate_report_from_findings() → LLM 综合分析 → 风险矩阵 + 修复路线图 11.编排层 状态写入 {analysis_result, pentest_findings, next_step:"plan_response"} 12.编排层 plan_response → 生成 P0-P3 响应时间线 13.编排层 generate_report → Reporter Agent 整合所有内容 → Markdown 报告 14.API 层 AlertResponse {analysis, response_plan, report} → JSON 返回客户端

七、面试关键词速查表

关键词项目中如何体现一句话说清
MCP 协议 3 个独立 MCP Server，stdio 通信，动态工具发现 "把渗透测试工具封装成标准化 MCP Server，Agent 通过统一协议调用"
LangGraph StateGraph + 6 节点 + 条件路由 + MemorySaver "用 StateGraph 显式编排多 Agent 协作，不是黑盒 AgentExecutor"
多 Agent 协作 4 个 Agent: Analyzer/Pentest/Responder/Reporter "职责分离，每个 Agent 只做一件事，system prompt 精炼不出错"
ReACT 范式 ThreatAnalyzer 使用 create_react_agent "模型交替思考和行动，适合需要多步调查的安全分析"
RAG ChromaDB + ATT&CK/CVE/OWASP 种子数据 "先检索相关知识注入上下文，减少 LLM 幻觉"
工具调用 MCP 工具 + VirusTotal + CVE API "两种工具集成方式：MCP 标准协议 + 传统 API 调用"
适配器模式 langchain_bridge.py 动态转换 MCP→LangChain Tool "适配器模式桥接 MCP 和 LangChain 两个不同接口标准"
mock 模式 MCP_MOCK_MODE 环境变量，内置完整模拟数据 "无需安装系统工具即可演示完整渗透测试流程"
双路径架构 /analyze (被动) + /pentest (主动) "同一套代码覆盖 SOC 研判和红队侦察两种安全运营场景"
FastAPI 异步端点 + Pydantic 验证 + 自动 Swagger 文档 "原生异步支持 LangGraph ainvoke，/docs 可直接调试 API"
