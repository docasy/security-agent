三、每个节点的输入输出 —— 状态在节点间的流转

LangGraph 的核心机制：每个节点接收完整状态 → 返回部分更新 → Graph 自动合并。 下面追踪一个 Dual Request（同时有告警和渗透目标）通过工作流时，状态如何变化。

State = { task_type: "alert" | "pentest", // ← 控制走哪条路 alert_data: str, // ← 路径 A 的输入 target: str, // ← 路径 B 的输入 messages: [...], // ← 对话历史 (add reducer 追加) rag_context: str, // ← 节点①写入，节点②读取 analysis_result: str, // ← 节点②写入，节点③④读取 pentest_findings: dict, // ← 节点②(渗透)写入 response_plan: str, // ← 节点③写入，节点④读取 final_report: str, // ← 节点④写入，最终输出 next_step: str, // ← 控制流字段 }

路径 A 示例：一条安全告警的完整旅程

请求: POST /analyze {"alert_data": "检测到来自 45.33.32.156 的异常登录...", "task_type":"alert"} 节点⓪ route_entry 读: task_type = "alert" 写: next_step = "retrieve_knowledge" 决策: task_type=="alert" → 走蓝队路径 节点① retrieve_knowledge 读: alert_data = "检测到来自 45.33.32.156 的异常登录..." 动作: kb.query(alert_data, k=3) → ChromaDB 检索 TOP-3 相关文档 写: rag_context = "[来源: MITRE ATT&CK] Phishing is the most common initial access vector (T1566)..." "[来源: Incident Response] Ransomware attack lifecycle: 1) Initial access..." 节点② analyze_threat 读: alert_data + rag_context 动作: ReACT Agent 循环 → Step 1 (思考): "提取 IOC... IP 45.33.32.156" Step 2 (行动): query_ip_reputation("45.33.32.156") → VT 查杀 Step 3 (观察): "malicious:15, suspicious:3 → 确认为恶意IP" Step 4 (思考): "搜索关联漏洞..." Step 5 (行动): search_cve("SSH brute force") → CVE 搜索 Step 6 (结论): "研判: Confirmed Threat, 建议立即封禁IP" 写: analysis_result = "研判结论: 确认威胁... IOC: 45.33.32.156 (恶意)..." 写: next_step = "plan_response" 节点③ plan_response 读: analysis_result 动作: Responder.generate_response_plan() → LLM 生成分阶段计划 写: response_plan = "1. 立即封禁 IP 45.33.32.156 (自动化)\n2. 检查受影响账户..." 写: next_step = "generate_report" 节点④ generate_report 读: alert_data + analysis_result + response_plan 动作: Reporter.generate_report() → LLM 整合为结构化 Markdown 写: final_report = "# 安全事件分析报告\n## 1. 事件概述\n..." 写: next_step = "done" → 返回 AlertResponse {analysis, response_plan, report} → JSON 响应

路径 B 示例：一次渗透测试的完整旅程

请求: POST /pentest {"target": "192.168.1.1", "task_type":"pentest"} 节点⓪ route_entry 读: task_type = "pentest" 写: next_step = "run_pentest" 决策: task_type=="pentest" → 走红队路径，跳过 RAG 节点② run_pentest 读: target = "192.168.1.1" 动作: PentestAgent.run_with_mcp_tools(target) → ┌─ Step 1: mcp.call_tool("nmap","scan_ports") → 发现 22,80,443,3306,8080 开放 ├─ Step 2: mcp.call_tool("nmap","detect_services")→ Apache 2.4.41, MySQL 5.7.38 ├─ Step 3: mcp.call_tool("nmap","vuln_scan") → SWEET32 确认, Log4Shell 待验证 ├─ Step 4: mcp.call_tool("whatweb","fingerprint") → WordPress 5.8.3, PHP 7.4 └─ Step 5: mcp.call_tool("exploitdb","search") → 搜索 "Apache 2.4.41" 漏洞利用 动作: pentest.generate_report_from_findings() → LLM 综合所有发现 写: pentest_findings = {ports:"...", services:"...", ...} ← 原始数据 写: analysis_result = "### 渗透测试综合评估\n风险等级: High\n..." ← LLM 报告 写: next_step = "plan_response" 节点③ plan_response 读: analysis_result (包含渗透测试的综合发现) 动作: Responder 检测到是渗透测试结果，调整 prompt 措辞 写: response_plan = "P0: 升级 OpenSSH + MySQL 到最新版..." 节点④ generate_report 读: target + analysis_result + response_plan 写: final_report = "# 渗透测试评估报告\n## 1. 风险评估总结\n..."

┌─────────────────────────────────────────────────────────────────────────────┐ │ 四个 Agent · 一条生产线 │ │ │ │ ┌───────────────────────┐ │ │ │ ThreatAnalyzer │ 职责：理解告警 → 提取IOC → 调VT/CVE → 研判 │ │ │ (蓝队专属) │ 范式：ReACT (思考⇄行动循环) │ │ │ │ 工具：query_ip / query_hash / search_cve │ │ │ 输入: alert_data │ 输出：analysis_result (研判结论) │ │ │ + rag_context │ │ │ └───────────────────────┘ │ │ │ │ ┌───────────────────────┐ │ │ │ PentestAgent │ 职责：协调 3 个 MCP Server 执行渗透侦察 │ │ │ (红队专属) │ 范式：分步编排 (程序化 + LLM 驱动双模式) │ │ │ │ 工具：nmap / whatweb / exploitdb (MCP) │ │ │ 输入: target │ 输出：pentest_findings (原始) + analysis_result (报告)│ │ └───────────────────────┘ │ │ │ │ ┌───────────────────────┐ │ │ │ IncidentResponder │ 职责：根据分析结果生成分阶段响应计划 │ │ │ (两条路径共享) │ 范式：Prompt-driven (结构化 prompt 模板) │ │ │ │ 关键：区分自动化操作 vs 需人工确认 │ │ │ 输入: analysis_result│ 输出：response_plan (P0/P1/P2/P3 时间线) │ │ └───────────────────────┘ │ │ │ │ ┌───────────────────────┐ │ │ │ ReportGenerator │ 职责：整合所有信息 → 结构化 Markdown 报告 │ │ │ (两条路径共享) │ 范式：Prompt-driven (报告模板驱动) │ │ │ │ 关键：技术层 (细节) + 管理层 (影响范围) 双受众 │ │ │ 输入: alert/target │ 输出：final_report (结构化 Markdown) │ │ │ + analysis │ │ │ │ + response │ │ │ └───────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────────────────────────┘
面试要点：为什么 Analyzer 用 ReACT 而 Responder/Reporter 只用 Prompt？
Responder 和 Reporter 是纯文本转换任务——输入一段文本，输出另一段文本，不需要调用外部工具。
Analyzer 需要多步调查——查 IP → 看结果 → 决定是否查 CVE → 综合判断，ReACT 的思考-行动循环更适合。
这种因地制宜的范式选型，证明你不是"所有 Agent 都用同一种模式"的初学者。
